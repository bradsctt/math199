Part 1: Pixels

What does a computer actually see?

To a computer, an image is a grid of numbers. Each number is a pixel: a brightness value from 0 (black) to 255 (white). Click any pixel below to change it.

The maths behind pixels
  • A greyscale image is a 2D matrix \(I\) where each entry \(I[i,j] \in \{0, 1, \ldots, 255\}\): \(I \in \mathbb{Z}^{H\times W}\).
  • A colour image has three such matrices, one per channel: \(I_{\text{colour}} \in \mathbb{Z}^{H\times W\times 3}\) → [R, G, B] per pixel.
  • A 1080p image (1920×1080) stores \(1920 \times 1080 \times 3 =\) 6,220,800 values.
Part 2: Filters & Convolution

Image filters:
Edges, blur & morphology

A filter transforms each pixel by combining it with its neighbours. Different filter families do very different things: edge-detection filters (like Sobel) highlight where brightness changes sharply; smoothing filters (Gaussian blur) reduce noise by averaging neighbours; morphological filters (erosion, dilation) reshape the regions of an image by expanding or shrinking them.

Until roughly 2012, this was how computer vision worked: computer scientists and mathematicians designed filters by hand to perform specific tasks.
Deep learning changed that. Instead of designing filters, we let the network work out its own numbers by learning from its mistakes.
The next section shows how that learning works.
Convolution and beyond
  • Convolution slides a kernel over every position: \(O[i,j] = \sum_m \sum_n I[i{+}m,\, j{+}n] \cdot K[m,n]\). All linear filters (Sobel, blur, sharpen) follow this formula — only the kernel numbers change.
  • Sobel edge detection — \(G_x = [[-1,0,1],\,[-2,0,2],\,[-1,0,1]]\) approximates \(\partial I/\partial x\) (vertical edges); \(G_y\) captures \(\partial I/\partial y\) (horizontal edges). Combined edge magnitude: \(\sqrt{G_x^2 + G_y^2}\) at each pixel.
  • Gaussian blur — weights follow a 2-D Gaussian bell curve, so nearby pixels contribute more than distant ones. Smooths out small speckly noise while keeping the overall shapes in the image intact.
  • Erosion and dilation work differently from the filters above — instead of a weighted sum, they just look at the neighbouring pixels directly.
    Erosion: \(E[i,j] = \min_{(m,n)\in S} I[i{+}m,\, j{+}n]\) — replaces each pixel with the darkest one nearby, shrinking bright regions.
    Dilation: \(D[i,j] = \max_{(m,n)\in S} I[i{+}m,\, j{+}n]\) — replaces each pixel with the brightest one nearby, expanding bright regions.
Part 3: How AI Learns

Weights, activations,
and gradients

A trained neural network contains thousands of filters like the one you just made, but nobody designed any of them. Each number is found by repeating one loop: provide an input, measure how wrong the output is (error calculation), then adjust every filter a little bit to reduce the error. Below, we'll cover the terms that loop relies on, then step through it on a small example network.

But first, let's introduce some terminology:

  • Node and layer: a node is one small unit that computes a weighted sum and passes it on. A layer is a group of nodes at the same stage within a neural network. The tiny network we will look at today has three layers: an input layer (\(x\)), a hidden layer (\(h_1\)), and an output layer (\(\hat{y}\)). Real networks stack many more hidden layers between input and output, which is what "deep" in deep learning refers to.
  • Weight, like \(w_1\) or \(w_2\) below: a number the network multiplies its input by - this performs the same job as the nine numbers in the Sobel kernel above. Training only ever changes these weights (and a similar number called a bias, \(b_1\) and \(b_2\)), but never the network's structure.
  • \(z\): the raw weighted sum at a node, e.g. \(z_1 = w_1 \times x + b_1\). An activation function is then applied to \(z\) to produce the node's output \(h = f(z)\). In the network below we apply ReLU after the hidden node, so \(h_1 = \text{ReLU}(z_1)\). The output node skips this step, so \(\hat{y} = z_2\) directly.
  • \(\hat{y}\) ("y-hat"): the network's prediction, and \(y\), the true answer it's trying to match.
  • Loss: a single number measuring how wrong \(\hat{y}\) is. Bigger means worse; zero means a perfect prediction.
  • Gradient, written \(\partial L/\partial w\): the answer to one question about one weight, "if I adjust this weight slightly, does the loss go up or down, and by how much?" That's the number we need for every weight in the network.
  • \(\alpha\) (learning rate): how big a step we take once we know a gradient. This is typically a small number; in the example below we use 0.01.

One of those terms is worth a closer look before we see it in action: the activation function. Here's why it matters, then we'll step through the example network.

Activation functions

If every layer only multiplies and adds, stacking more of them doesn't add any real power — mathematically, it's no different from having just one layer, no matter how many you stack. An activation function \(f\) is applied to each raw sum \(z\) before it's passed forward, giving \(h = f(z)\). This small extra step is what lets deep networks model curves, boundaries, and complex patterns that a single straight-line layer never could.

ReLU — used in almost all modern CNNs
\(\text{ReLU}(z) = \max(0,\,z)\)
\(\text{ReLU}'(z) = \begin{cases}1 & z > 0\\0 & z \leq 0\end{cases}\)
Sigmoid — used for probability outputs
\(\sigma(z) = \dfrac{1}{1+e^{-z}}\)
\(\sigma'(z) = \sigma(z)\,(1-\sigma(z))\)
Drag to explore — \(z =\) 1.0
ReLU(z) = 1.00   ReLU′(z) = 1
\(\sigma\)(z) = 0.731   \(\sigma'\)(z) = 0.197
Each activation's derivative feeds into the chain rule during training.
For ReLU: the derivative is 1 when \(z > 0\) (gradient passes through unchanged) and 0 when \(z \leq 0\) (gradient is zero — dead neuron).
For sigmoid: the derivative peaks at 0.25 when \(z=0\) and shrinks toward 0 at both extremes, causing gradients to vanish across many layers.
ReLU avoids this, which is why it dominates in deep networks.

Forward pass: compute the prediction
\(z_1 = w_1 \times x + b_1 = 3 \times 2 + 1 = 7,\quad h_1 = \text{ReLU}(z_1) = \max(0,\,7) = 7\)
\(z_2 = w_2 \times h_1 + b_2 = -2 \times 7 + 4 = -10,\quad \text{so}\quad \hat{y} = -10\)
\(L = \tfrac{1}{2}(y - \hat{y})^2 = \tfrac{1}{2}(10-(-10))^2 =\) 200

The target was \(y=10\) and the network predicted \(\hat{y}=-10\), so \(L\) squares that gap (and halves it, a convention that keeps the derivative tidy). A perfect prediction would give \(L=0\).

←
Backward step 1: how sensitive is the loss to \(\hat{y}\)?

Before touching any weight, we first ask: if \(\hat{y}\) itself had been slightly different, how much would \(L\) have changed? That sensitivity is \(\partial L/\partial\hat{y}\), the gradient of the loss with respect to the prediction.

\(\dfrac{\partial L}{\partial\hat{y}} = -(y - \hat{y}) = -(10-(-10)) =\) −20

This is the starting gradient. Every weight's gradient below is this number, chained through the layers between it and \(\hat{y}\).

←
Backward step 2: the gradient for \(w_2\)

Chain rule: \(\dfrac{\partial L}{\partial w_2} = \dfrac{\partial L}{\partial\hat{y}} \cdot \dfrac{\partial\hat{y}}{\partial z_2} \cdot \dfrac{\partial z_2}{\partial w_2}\)
Since \(z_2 = w_2 \cdot h_1 + b_2\), we get \(\partial z_2/\partial w_2 = h_1 = 7\). And \(\partial\hat{y}/\partial z_2 = 1\) because the output node skips the activation step, so \(\hat{y} = z_2\) directly.

\(\partial L/\partial w_2 = -20 \times 1 \times 7 =\)
←
Backward step 3: the gradient for \(w_1\), one layer deeper

The chain now passes through \(w_2\) and \(h_1\) to reach \(w_1\):
\(\dfrac{\partial L}{\partial w_1} = \dfrac{\partial L}{\partial\hat{y}} \cdot \dfrac{\partial\hat{y}}{\partial z_2} \cdot \dfrac{\partial z_2}{\partial h_1} \cdot \dfrac{\partial h_1}{\partial z_1} \cdot \dfrac{\partial z_1}{\partial w_1}\)
Each factor: \(\partial z_2/\partial h_1 = w_2\) (since \(z_2 = w_2 \cdot h_1 + b_2\)), \(\partial h_1/\partial z_1 = \text{ReLU}'(z_1) = 1\) (since \(z_1 = 7 > 0\)), \(\partial z_1/\partial w_1 = x\) (since \(z_1 = w_1 \cdot x + b_1\)).
\(= -20 \times 1 \times w_2 \times 1 \times x = -20 \times (-2) \times 2\)

\(\partial L/\partial w_1 = -20 \times 1 \times (-2) \times 1 \times 2 =\)
↻
Weight update: step downhill with \(\alpha = 0.01\)
\(w_{2,\text{new}} = -2 - 0.01 \times (-140) = -2 + 1.4 =\) −0.6
\(w_{1,\text{new}} = 3 - 0.01 \times 80 = 3 - 0.8 =\) 2.2

Repeated across thousands of examples, \(\hat{y}\) gradually approaches \(y\). Training a network is this loop, run millions of times.

Step 0 / 5
Why does this make the model better?
  • Each gradient answers: "if I increase this weight, does the loss go up or down?"
  • We move the weight a small step in the opposite direction, downhill, scaled by the learning rate \(\alpha\): \(w_\text{new} = w_\text{old} - \alpha \times \partial L/\partial w\).
  • Done millions of times across thousands of examples, the model's predictions steadily improve.

Part 5: Object Detection, Segmentation & Pose

Detection, segmentation and pose

The network you just trained is quite simple: it looks at a whole image and picks one label for it, with no idea where an object is or how many there are. Real-world computer vision usually needs more than that - detection draws a box around each object, segmentation labels every pixel, and pose estimation finds the joints of each person.

How detection, segmentation, and pose work
  • Earlier models scanned the image in thousands of small windows. Modern detectors like YOLO do it in one pass, dividing the image into a grid and predicting boxes and classes for every cell at once: output per cell is [x, y, w, h, confidence, class_1, ..., class_80]. The browser demo here uses YOLOS-tiny, with about 6.5 million parameters compared to the roughly 400,000 your CIFAR model just trained.
  • A bounding box says there is a bus somewhere in this rectangle. A segmentation mask says exactly which pixels belong to the bus, precision that matters for self-driving cars (where exactly is the road edge?), medical imaging (where exactly is the tumour boundary?), and photo editing (cut out the person, keep the background).
  • Pose models predict 17 keypoints per person (eyes, ears, nose, shoulders, elbows, wrists, hips, knees and ankles), then connect them into a skeleton: Person = [(x₁,y₁,conf₁), ..., (x₁₇,y₁₇,conf₁₇)]. This powers gesture control, sports analysis, physiotherapy apps, and animation motion capture.
Part 6: 3D Vision AI

Seeing depth: Adding a third dimension

Everything so far has treated images as flat. But the real world isn't flat: determining depth - or an object's distance from a camera - is another key part of understanding the world.

There are a few ways to measure it: LiDAR, radar, stereo cameras (like human eyes), and, more recently, monocular depth estimation - using a single ordinary photo. In the example below, a neural network does exactly that: estimating the distance for every pixel in one image. The result is a depth map and a point cloud you can spin around.

How depth estimation works, and how it connects to LiDAR
  • Humans judge depth with one eye using several cues: distant objects appear smaller, are overlapped by nearer ones, and have softer edges. Depth Anything is a neural network trained on over 60 million images to learn these patterns: input H×W×3 image → encoder (feature extraction) → decoder → H×W depth map.
  • Under the hood, this is built the same way as ChatGPT, Claude, and other AI - but instead of predicting the next word in a sentence, it predicts a depth value for every pixel. Point cloud conversion: for each pixel (u,v) with depth d, the 3D position is roughly (u·d, v·d, d), simple geometry, but powerful when you have depth for every pixel.
  • Camera-based depth estimation and LiDAR solve the same problem differently. LiDAR fires laser pulses and measures the return time: direct and precise, but expensive and sparse. Cameras are cheap and dense but ambiguous about absolute scale.
  • Modern autonomous systems often fuse both: LiDAR gives accurate sparse depth, the camera fills in dense texture. This is exactly the pipeline used in self-driving cars, and increasingly in drones for navigation and mapping.
